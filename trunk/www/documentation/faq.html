1. General

Q. What is JDIC?
A. JDIC is short for JDesktop Integration Components. The project aims to make Java technology-based applications (Java applications) first-class citizens of current desktop platforms without sacrificing platform independence. The JDIC project home page is: https://jdic.dev.java.net.

Q. When was JDIC released?
 A. The JDIC project was first released on 1 June 2004. The code is available via CVS, and periodic binary releases. 

Q. What is the current version of JDIC?
A. JDIC source is continually available via CVS, and the latest versioned binary release is available here. 

Q. Under which license terms was JDIC released?
A. JDIC is released under LGPL. Take a look at JDIC's COPYING file for detailed license terms.

Q. What are the supported platforms?
A. Briefly, Windows, Linux and Solaris. Refer to the JDIC System Requirements for specific versions and distributions. Some contributors provided patches and binaries for limited support for Mac OS X and FreeBSD platforms. Refer to the Mac OS X and FreeBSD support discussion threads for more information. The incubator projects may have different platform support matrixes, refer to the JDIC Incubator Projects space for their specific system requirements.

Q. Will there be a port to Mac?
A. JDIC currently doesn't fully support Mac. Some contributors provided patches and binaries for limited support on Mac OS X. There are still gaps waiting to be closed. We'd very much welcome any volunteers or effort to port JDIC to Mac.

Q. Isn't Swing enough? Why are you introducing yet another desktop toolkit?
A. JDIC is not a replacement for Swing/AWT (that is, a GUI widget library). It's a set of APIs and tools to integrate with system services. It's compatible with Swing: you can use the components under Swing or AWT. It helps to make Java a more complete platform by providing new APIs that gives portable access to some useful system services.

Q. What are the incubator projects?
A. Some projects are associated with JDIC but aren't part of the source or binary distribution of the JDIC project. These incubator projects are released as separate downloads and develop at their own speed with different quality criteria. Once an incubator project complies with the requirements of JDIC, it might move into the JDIC project.

Q. Can I add an incubator project? How?
A. Yes. We consider the services currently available in JDIC a starting point rather than complete. We'll welcome efforts to address other desktop integration aspects. If you have a relevant project and want to be moved into JDIC someday, you could submit an incubator project request to jdic@jdic.dev.java.net. If it's approved, it could be hold in the JDIC Incubator Projects space.

Q. Is there any Java package naming convention?
A. Yes. All the Java packages of the JDIC project, including the incubator projects, use the top package name prefix org.jdesktop.jdic. The first package name segment after org.jdesktop.jdic is generally the component or incubator project name. The following package names are used in the early release:

  • org.jdesktop.jdic.desktop - for desktop APIs.
  • org.jdesktop.jdic.filetypes - for file types APIs.
  • org.jdesktop.jdic.browser - for browser APIs.
  • org.jdesktop.jdic.packager - for JDIC Packager APIs.
  • org.jdesktop.jdic.screensaver - for SaverBeans Screensaver SDK (incubator project).
  • org.jdesktop.jdic.tray - for Tray Icon API (incubator project).

2. Downloading

Q. Where can I get a binary?
A. JDIC can be fetched as a binary in a versioned release. You could find the JDIC binary downloads, including the incubator projects downloads here. 

Q. Where can I get the source?
A. There are several ways you can access the JDIC source code:

  • Browse
    Browse the up-to-the-minute latest version of the source code online.
  • Download
    • FTP - Download the source zip file for a specific release.
    • CVS - The latest source could be checked out using CVS following the "CVS client setup instructions" here. It lets you get up-to-the-minute changes. For each JDIC release, a tag is created to help people pull the right source. The tag names are of the form  "JDIC_major_minor_micro". For example, JDIC release 0.8.4 has a tag "JDIC_0_8_4".

3. Installation and Configuration

Q. Why is there a defmailer.properties file in the binary distributions for Linux/Solaris?
A. This property file is used to define the default mail program path. While the user uses Desktop.mail() method, it tries to find the system default mail program in the GConf database. If the default mail program is not set, the method looks in this property file for the default mail program path.

Q. Is a Mozilla binary required to use package org.jdesktop.jdic.browser?
A. Yes for Linux/Solaris, and yes for Windows if Mozilla is the embedded browser. Since the JDIC org.jdesktop.jdic.browser package makes use of some Mozilla code, a Mozilla binary is required to use the package. For Linux/Solaris platforms, since Mozilla is always the embedded browser, a Mozilla binary is always required to use the package. For Windows platforms, the default browser (Internet Explorer or Mozilla) is embedded, if the Internet Explorer is embed, no Mozilla binary is required to use the package.

4. Building the Source

Q. Can I pull a copy of the source and build my own binary?
A. Yes. After you pull the source code, follow the "Build the Source" guidelines to build your own binary. We like to provide up to date build documentation. If you find something to be out of date, incorrect, or missing altogether, report a bug using JDIC Issue tracker, or send a mail to jdic@jdic.dev.java.net.

Q. What type of build system does JDIC use?
A. JDIC uses Apache Ant as the build system on all platforms.

Q. Can I build the JDIC source without building all the Mozilla stuff?
A. No. The JDIC org.jdesktop.jdic.browser package makes use of some Mozilla code, some native code won't compile unless the Mozilla stuff it depends on has been built. The other parts don't depend on Mozilla code.

Q. Why the Microsoft Platform Software Development Kit (SDK) Update is needed to build the source on Windows platforms?
A. The SDK Update is needed for below reasons:

  • The JDIC API native JNI code on Windows uses a Windows Shell function AssocQueryString to search for and retrieve a file association-related string from the registry. This functions is supported in the Platform SDK Update.
  • The JDIC Packager on Windows uses the latest Windows MSI SDK provided by the SDK Update.
  • The JDIC Packager on Windows uses a sample MSI template contained in the SDK Update. The sample template is used as a raw MSI template to be modified to generate the final MSI file.
  • The JDIC Packager tool jnlp2msi uses two tools provided by the SDK Update: msidb.exe and makecab.exe.

6. Troubleshooting

Q. When building the source, I get the error message "Makefile:42: /config/autoconf.mk: No such file or directory." on Unix, and " [exec] ...\msgserver.cpp(23) : fatal error C1083: Cannot open include file: 'prlog.h': No such file or directory" on Windows. Why?
A. The environment variable MOZILLA_SRC_HOME is not set to the Mozilla source directory including a built binary. Refer to the JDIC Build the Source instructions.

Q. When building the source, I get the error message "GnomeBrowserService.cpp:21:17: jni.h: No such file or directory" on Unix. Why?
A. The environment variable JAVA_HOME is not set to the J2SDK home directory.
