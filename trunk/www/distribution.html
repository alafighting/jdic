Build A JDIC Distribution

This is the page for the JDIC Release Engineer to build a JDIC distribution and make it public. Below are the detailed steps.
  1. Update the version and release information.
    For a release with a new major or minor number, update below version and release information in the CVS repository.
    • Update the "version" and "release" property values in <JDIC source directory>/jdic/build.xml, <JDIC source directory>/packager/build.xml files.
    • Update the "PRODUCT_VERSION" and "PRODUCT_RELEASE" values in <JDIC source directory>/packager/src/unix/jnlp2rpm and <JDIC source directory>/packager/src/solaris/jnlp2pkg script files. And update the "MY_FULL_VERSION" and "MY_RELEASE_NO" values in <JDIC source directory>/packager/src/win32/script/jnlp2msi.cpp file.
    • Update the <JDIC source directory>/ChangeLog file for the latest bug fixes and updates. The bug fixes after the last release could be checked from Issue Tracker.
    • Update the "Version" and "Release" information in <JDIC source directory>/README.html file, and include any other changes.

  2. Build the source.
    Follow the Build the Source instruction to build the source, with a binary release is generated under dist/<platform> directory.

  3. Compile the demo applications.
    Make sure the demo applications under dist/<platform>/demo directory are compiled from source files (.java) into binaries (.class). The released demo applications should include both the source files (.java) and the compiled binaries (.class).

  4. Create a release tag in the CVS repository.
    Before creating a release tag, make sure all the changes to the CVS repository are checked in. The release tag should be in the form "JDIC_major_minor_micro", such as "JDIC_0_8_4". If necessary, create a branch with the branch name in the form "JDIC_major_minor_micro_BRANCH", such as "JDIC_0_8_4_BRANCH".

  5. Create a binary package.
    Create a binary zip (or tarball, msi for Windows, rpm for Linux, pkg and Solaris) package with the <JDIC source directory>/dist/<platform> directory, using command "zip -r <package name>.zip <JDIC source directory>/dist/<platform>". The binary package name should be in the form "jdic-major.minor.micro-bin-<platform>.zip", such as "jdic-0.8.4.1-bin-windows.zip".

  6. Test.
    Test the binary package to make sure it works. Check the included demo applications for a preliminary test.
    A complete test should start from the a complete binary package (in the form "jdic-major.minor.micro-bin-<platform>.zip"), and include below tests:
    • The package format:
      - the package can be unzipped or installed correctly.
    • The documentation:
      - the README.html file gives all the neseasary and udpated instructions. The test should base on this file.
      - javadoc is included, in the right format and updated.
      - the ChangeLog file contains the latest updates/bug fixes.
      - the COPYING file is included.
    • The demo applications under <JDIC binary directory>/demo directory.
      - all the demo applications include both the source (.java) and binary (.class) files.
      - all the demo application binaries work.
    • API test.
      - all the test cases work.

  7. Create a source package.
    Create a source zip (or tarball) package with source files with the release tag. One approach is to check out the JDIC source with the release tag to an empty local folder, then create the zip (or tarball) package with that folder. The source package name should be in the form "jdic-major.minor.micro-src.zip", such as "jdic-0.8.4-src.zip".

  8. Distribute.
    Make the created binary and source packages downloadable at the JDIC Documents & files page. If the distribution has a new major or minor release number, create a separate download directory in the form "jdic-major.minor.micro", such as "jdic-0.8.4". If the distribution only has a new micro release number, put the packages under the download directory with the same major and minor release numbers.

  9. Announce.
    Announce the new distribution to the public:
    • Give an announcement to the JDIC Forum.
    • Give an announcement to the mailing list announce@jdic.dev.java.net, which is for the announcements of new releases and other major project events.
    • Give an announcement to the Java Product Announcements page.
    • Give announcements to somewhere else if necessary.

  10. Update the JDIC website.
    • Add the new release tag or branch name created for this release to below CVS tags table.
    • Update the online WebStart demos on the JDIC homepage.
    • If javadoc for the release is updated, update the javadoc link on the JDIC homepage.
    • Update other documentations on the JDIC website if necessary.
